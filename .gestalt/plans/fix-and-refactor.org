* DONE [#A] Harden allocator invariants and public inclusion model
Effort: 1 day
Goal: Eliminate parameter combinations and integration patterns that currently lead to corruption or link failures.
Notes: Start here because these issues affect every downstream consumer. Keep the public API names stable. Prefer the smallest change that preserves the header-only design.

** DONE [#A] Reject impossible pool geometries at init time
Why: The free list stores a full pointer inside each free block, so any block smaller than =sizeof(void*)= can corrupt adjacent memory. A zero =nmemb= also underflows the free-list initialization loop for the last block. The current init path only checks “power of two”, which is not sufficient.
Change: Update =sfpool_init()= to reject =nmemb == 0=, reject any =blocksize < sizeof(void*)=, and keep the existing power-of-two rule. While touching this logic, verify that the computed pool size cannot silently wrap when multiplying =nmemb * blocksize=. Return =0= on all invalid configurations and leave the pool in a clearly failed state.
Tests: Add deterministic checks that =sfpool_init()= returns =0= for =nmemb == 0= and for undersized power-of-two blocks such as 1, 2, and 4 on 64-bit builds. Keep one passing case to confirm valid parameters still initialize correctly.
Done when: Invalid pool geometries fail before touching allocator state, valid geometries still initialize, and the new checks run in the local test executable.

** DONE [#A] Make header-only inclusion safe across multiple translation units
Why: =sfpool.h= contains full function definitions with external linkage. Including it from more than one C file will create duplicate symbols at link time, which defeats the “header-only” claim for real projects.
Change: Choose one minimal model and apply it consistently. Preferred options are: 1) convert the implementation functions to =static inline= where valid, or 2) adopt a single-header implementation pattern with declarations by default and definitions only under an opt-in macro such as =SFPOOL_IMPLEMENTATION=. Keep the public API stable and document the chosen inclusion model in the header comment and README.
Tests: Add a compile-and-link smoke test with two tiny translation units that both include =sfpool.h=. The test only needs to prove that the selected header-only model works; it does not need runtime assertions.
Done when: Two separate C files can include =sfpool.h= and link cleanly, and the intended inclusion model is explicit in code and docs.

* WIP [#A] Close remaining allocator correctness gaps
Effort: 1 day
Goal: Remove the remaining undefined behavior paths in =malloc/free/realloc/zero= without broadening the design.
Notes: Keep behavior unsurprising for existing callers. If any branch semantics change, update the relevant Doxygen comments in the same L2.

** DONE [#A] Make pool-to-heap realloc failure-safe
Why: When reallocating a pool-owned block to a larger size, the code calls =malloc(size)= and immediately copies into the result. If =malloc()= fails, the current path dereferences =NULL= and can also lose the original allocation semantics.
Change: In the pool-to-heap branch of =sfpool_realloc()=, check the result of =malloc()= before copying or releasing the old pool block. On failure, return =NULL= and leave the original block untouched so the caller can still free or reuse it. Only update profiling counters on a successful transition.
Tests: Add a deterministic failure case by requesting an impossibly large size, e.g. =SIZE_MAX=, from a pool-owned pointer. Assert that the call returns =NULL= and that the original pointer is still recognized by =sfpool_contains()= and remains safe to free.
Done when: A failed grow-realloc no longer crashes, preserves the original block, and the regression is covered by a test.

** DONE [#B] Zero trailing bytes in secure wiping
Why: =sfutil_zero()= currently clears only =size >> 2= 32-bit words. For sizes not divisible by four, the last 1-3 bytes remain untouched, which breaks the stated secure-zeroing intent.
Change: Replace the current implementation with a simple, portable full-range wipe that clears every byte. If retaining the word-at-a-time fast path, add a tail pass for the remainder. Update the doc-comment so it accurately describes the implementation.
Tests: Add a small targeted test that fills a non-multiple-of-four buffer with non-zero bytes, calls =sfutil_zero()=, and asserts that every byte became zero.
Done when: Secure wiping clears the complete requested range and the test covers an odd-sized buffer.

** WIP [#B] Make FALLBACK semantics internally consistent
Why: =sfpool_malloc()= always falls back to system =malloc()=, while =sfpool_free()= and =sfpool_realloc()= only support heap-backed pointers when =FALLBACK= is enabled. The macro currently describes a mode that the code does not enforce consistently.
Change: Decide the contract first. Either make fallback unconditional and remove the conditional branches/macros, or fully honor =FALLBACK= across allocation, free, and realloc. Prefer the simpler contract if there is no real use case for disabling fallback. Update comments and README text to match the chosen behavior.
Tests: Run the default build and, if fallback remains configurable, add one compile/run case with fallback disabled to prove the behavior is coherent.
Done when: Allocation, free, and realloc agree on fallback ownership rules, and the documented behavior matches the implementation.
